# 技术栈选型说明

## 一、选型原则

这个项目的第一阶段不是追求“技术最全”，而是追求下面三点：

1. Windows 本地能稳定跑起来
2. 面试时能讲清楚每一层为什么这样设计
3. 后续能平滑扩展到真实 LLM、更多工具和更复杂的观测能力

## 二、后端为什么选 FastAPI + SQLite

- FastAPI：接口定义清晰，调试成本低，适合快速验证调用链产品原型
- SQLite：避免一开始引入 Docker 和数据库服务，降低本地部署成本
- SQLAlchemy 2：保留后续切换到更正式数据库的空间
- Pydantic 2：让请求和响应结构更明确，适合做 trace 数据建模

## 三、前端为什么选 React + Vite + TypeScript

- React：组件化展示时间线、摘要卡片和筛选面板比较自然
- Vite：本地启动快，适合频繁迭代 UI
- TypeScript：Trace 结构字段很多，用类型可以减少前后端字段漂移

## 四、为什么当前不直接上更重的框架

- 暂不引入 LangChain：第一版更需要显式执行流，而不是被框架封装后的黑盒链路
- 暂不引入 Docker：当前主要目标是原生 Windows 可跑
- 暂不引入 Tailwind：当前 UI 复杂度不高，先把可观测性产品逻辑搭稳

## 五、后续可升级点

- 升级到真实 LLM SDK
- 增加 OpenTelemetry 埋点
- 接入更正式的数据库和迁移工具
- 增加前端图表和运行对比视图
