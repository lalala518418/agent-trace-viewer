# 开发工作流

## 一、工作流目标

这个项目不是只追求“代码能跑”，而是追求：

- 每次改动都能解释原因
- 每次改动都有本地验证
- 每次会话结束都留下可读报告

## 二、标准流程

1. 选择一个聚焦的功能切片
2. 修改代码或文档
3. 运行与改动范围对应的本地验证
4. 更新自动生成的会话摘要草稿
5. 结束会话时由 hook 写出最终报告

## 三、为什么要这样做

- 方便你回顾整个项目框架和设计演进
- 方便后续准备实习或面试时回忆为什么这样实现
- 方便把一次次零散修改积累成可复用的工程经验

## 四、当前配套机制

- 验证 hook：改动后要求先做本地验证
- 报告 hook：会话结束时生成 `docs/session-reports/` 下的报告
- 人工摘要草稿：位于 `docs/session-reports/drafts/`

## 五、当前占位点

- 占位：自动报告还不能精确列出逐文件 diff
- 占位：报告中的架构级总结仍需要人工补充
